9. HINS Z Code Carryover between Service

#413

1. BERT will respond to HINS “Z” codes more broadly than SIMS does today.  In particular, BERT will not provide coverage to an otherwise-health-eligible member if Finance has a Z code in place.  A Z code with an effective date on or before a member’s hire date will prevent them from enrolling in benefits.  Additionally, a Z code that arises during the course of a member’s employment will result in coverage being cancelled from that point forward.  (Note that SIMS currently implements this second “after-arising” rule and ignores Z codes on or before the hire date.)  The rules of when coverage will be stopped/started in response to changes in the Z status (e.g., F -> Z or Z -> F) will be as follows:
1. When Finance turns “on” the Z code, coverage will end based on the first 1st rule – if the effective date of the Z code is the 1st of a month, then coverage ends on the last day of the prior month; otherwise, coverage will end on the last day of the month of the Z code effective date (Examples: Z code effective 3/1/2024 => coverage end date 2/29/2024.  Z code effective 3/2/2024 => coverage end date 3/31/2024.)
2. When Finance turns “off” the Z code, coverage will start on the effective date of the new health-eligible code (i.e., on the effective date of the “F” or “P” that follows the “Z”).

39. Employee UI Elections After OE

#707

Background

The time between the closing of open enrollment and the end of the calendar year (year end period) will need special handling for the Employee UI. This is because there are two coverage periods in play: the current benefit year and next benefit year. The system needs to provide a way for the employee to make choices for BOTH periods. Normally, after open enrollment closes, that option is no longer available to employees.

We also need to consider the case of a new employee who does not yet have their account fully activated. Currently the system only allows the employee to select their Hire event coverage choice.

Currently, the employee would need to contact DRB to have the change made on their behalf.

The administrative UI does not have this problem in that there are no restrictions imposed on the change reason for the staff (i.e. Open Enrollment option is always available).

There are two new features that could provide a solution.

Auto Open Enrollment Coverage Choice Created

With this feature, when an active employee makes a coverage change during the Year End Period, at the final screen where they review and submit their election, an additional checkbox will appear (already checked) titled “Also apply these choices to next benefit year (2025)”.

When a new employee makes a New Hire choice, the system will not show the checkbox, but will just behave like it IS checked. This is because the new employee does not currently have any other change types provided.

When this box is checked, the backend should create save the current choice for the current event AND create an Open Enrollment Choice for the new benefit year. The creation of the OE Coverage Choice should probably should be done using a stored procedure because there may be complex logic involved if there are any plan differences between the current and next benefit year. There may also be differences in HFSA maximums (the smaller amount should apply).

Allow Open Enrollment if Year End Period Coverage Choice Exists.

With this feature, the back end will include the Open Enrollment change reason in the list of options if there is a Coverage Choice that starts within the Year End Period (may be earlier than the OE closing date?).

Health Flexible Spending Account (HFSA) Rule

Currently HFSA elections are NOT automatically carried over during open enrollment. An employee must make a HFSA election for the new benefit year. When doing a default operation, the HFSA will be 0. When doing a convert, the HFSA will be whatever the prior election within the same benefit year.

If the hire dates do not match between the employer supplied

If the given date isn't even in the same benefit month, use

If the hire date in employment doesn't match the pending

- For Employment Status Change and Not Same Prior Election Benefit Package - convert benefits from prior election

- Convert process - benefits are derived from a copy or prior benefits unless found in the Benefit Package Convert table for the old/new benefit package, old benefit option id and date
- Carryover process - benefits are derived from a copy or prior benefits unless found in the Benefit Package Convert table for the old/new benefit package, old benefit option id and date.
